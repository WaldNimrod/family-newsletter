1 סקירת מערכת

המערכת מורכבת מ-7 מודולים עצמאיים עם normalizers ברורים ביניהם. כל מודול אחראי לדומיין אחד. המערכת רצה על השרת הביתי ומפיצה תוכן סטטי לשרת nimrod.bio.

📋 M1 · PROFILES
פרופילי בני משפחה, תחומי עניין, מקורות מידע, העדפות תוכן. מתעדכן מ-Feedback loop, מ-Dashboard ומסשן Cowork.

📊 M6 · FEEDBACK + SUBMISSIONS
קליטת משוב + קליטת ידיעות מבני המשפחה (תמונות, וידאו, קישורים, טקסט, PDF). עיבוד ולמידה. עדכון פרופילים.
→ IN: webhook events, click data, survey replies, family submissions ← OUT: profile_adjustments, weekly_report, submission_items[]
🖥️ M7 · DASHBOARD
כלי בקרה ותובנות לשיפור המערכת. מעקב עלות טוקנים, ביצועים, ותמיכה בתהליך הפיתוח. משמש את המפתח (Claude) באותה מידה כמו את המנהל.
→ IN: all data stores, token_usage_log ← OUT: config changes, schedule changes, insights

2 סביבות עבודה

שלוש סביבות נפרדות: פיתוח, פרודקשן (שרת ביתי), ואירוח (שרת web). כל צד השרת (לוגיקה, cron, API, DB) רץ בשרת הביתי. שרת האירוח מחזיק רק קבצי HTML סטטיים.
💻 סביבת פיתוח (Laptop)
המחשב הנוכחי — סביבת פיתוח בלבד
  • עריכה וכתיבת קוד (Cowork / IDE)
  • הרצת בדיקות ו-mock runs
  • Push ל-GitHub
  • עדכון הגדרות ומקורות דרך Cowork session
  • אין cron, אין production data
🏠 שרת ביתי (Production)
Linux server — מריץ את כל הלוגיקה
  • Clone מ-GitHub (production branch)
  • Cron jobs: סריקה, בנייה, הפצה, משוב
  • SQLite database (כל הנתונים)
  • Claude API calls (סיכומים + עריכה)
  • WhatsApp webhook endpoint (FastAPI)
  • FTP upload ל-nimrod.bio
  • Log files + monitoring
🌐 שרת אירוח (nimrod.bio / upress.co.il)
WordPress hosting — משמש לאירוח סטטי בלבד
  • תיקיית משנה ייעודית (e.g. /newsletter/)
  • גישת FTP יעודית
  • קבצי צד לקוח בלבד (HTML, CSS, images)
  • אין צד שרת פעיל
  • nginx/Apache משרת קבצים סטטיים
  • כתובת ציבורית: nimrod.bio/newsletter/

3.5 מבני הנתונים (Normalizers)

NCI — Normalized Content Item (פלט Scanner → קלט Normalizer)

NCI { id: string // hash of URL url: string title: string source: { name, type, url, trust_score } published_at: ISO8601 fetched_at: ISO8601 language: "he" | "en" raw_text: string // original summary / excerpt tags: string[] image_url: string? media: { type, url, thumbnail }? // for video/image submissions content_hash: string // for dedup across days is_submission: boolean // true if sent by family member submitted_by: string? // member_id if submission }

Family Submission (WhatsApp → M6 → NCI)

FamilySubmission { id: string // unique ID member_id: string // who sent it timestamp: ISO8601 message_type: "text" | "image" | "video" | "document" | "link" content: { text: string? // message text / caption media_url: string? // WhatsApp media URL (downloaded + stored) link_url: string? // extracted URL file_name: string? // PDF / doc filename mime_type: string? } status: "received" | "edited" | "published" edited_nci: NCI? // the processed version ready for edition edition_date: YYYY-MM-DD // which edition it appeared in }

NEO — Normalized Edition Object (פלט Normalizer → קלט Renderer)

NEO { date: YYYY-MM-DD family_name: string greeting: string // generated daily greeting family_content: [{ // submissions from family members submitted_by: string // display name headline: string // AI-edited headline summary: string // AI-edited summary media: { type, url }? // image/video if present original_nci: NCI }] member_sections: [{ member_id: string display_name: string // "יויו", "Shaked", etc. icon: string subtitle: string language: "he" | "en" | "both" articles: [{ nci: NCI // original item reference headline: string // generated headline summary: string // generated summary (language-aware) category: string // matched topic name score: number // relevance 0-100 }] }] discovery: [{ nci: NCI from_member: string to_member: string bridge_text: string // "נימרוד → צליל: ..." }] trivia: { history: string // today in history puzzle: string // daily puzzle (for צליל) answer: string // yesterday's answer } metadata: { items_fetched: number items_selected: number sources_scanned: number submissions_count: number generation_model: string tokens_used: { input, output, total, cost_usd } build_duration: number // ms } }

Feedback Event

FeedbackEvent { member_id: string timestamp: ISO8601 type: "rating" | "click" | "text" | "share" | "time_on_page" | "submission" newsletter: YYYY-MM-DD value: string // "1"-"4", URL clicked, free text, seconds article_id: string? // which article was interacted with }

4 תזמונים ותהליכים

שלושה מחזורים: יומי, שבועי, חודשי. שעת שליחה: 12:00 שעון ישראל. כל הזמנים ניתנים לשינוי — Phase 1 דרך Cowork, Phase 2 דרך Dashboard.
תהליך מחזור זמן (IST) מודולים מטרה
Newsletter Build יומי 09:00 M2→M3→M4 איסוף, עיבוד, שילוב ידיעות משפחתיות, בניית HTML
Newsletter Send יומי 12:00 M5 FTP upload ל-nimrod.bio + שליחת WhatsApp
Feedback Survey יומי 21:00 M5→M6 שליחת סקר מהיר + קליטת תשובות
Click Analytics יומי 23:00 M6 עיבוד קליקים ו-analytics מהיום
Submissions Intake אירוע 24/7 (webhook) M6 קליטת ידיעות מבני משפחה בזמן אמת, אישור קבלה
Source Refresh שבועי שישי 02:00 M2→M1 בדיקת תקינות מקורות, חיפוש מקורות חדשים, עדכון trust scores
Profile Optimization שבועי שישי 03:00 M6→M1 ניתוח משוב שבועי, התאמת ניקוד מקורות, שינוי priorities
Content Dedup Deep Clean שבועי שישי 04:00 M3 ניקוי כפילויות מתקדם (semantic similarity), עדכון content_hash index
Monthly Report חודשי 1 לחודש M6→M7 דו"ח חודשי: מגמות, שביעות רצון, עלויות, מה עניין, מה לא
Family Conversation חודשי 1 לחודש (מומלץ) M6→M1 סשן שיחה עם לפחות בן משפחה אחד — סקירה, רצונות, שיפורים. מומלץ, לא חוסם.
Deep Profile Review חודשי 1 לחודש M6→M1 סקר מעמיק ("מה תרצו לשנות?") + עדכון profiles מלא
Archive Maintenance חודשי 1 לחודש M3 vacuum DB, generate embeddings for old content, export monthly snapshot
שינוי תזמונים: Phase 1 — עריכת settings.json דרך סשן Cowork (עריכה, commit, deploy). Phase 2 — שינוי דרך Dashboard UI עם שמירה ישירה ל-config ועדכון cron.
איזון תוכן: התהליך השבועי מוודא שלא נחזור על תכנים. ה-Normalizer מחזיק index של content_hashes מ-30 יום אחרונים + semantic similarity check (embeddings) למניעת כפילויות רעיוניות.

5 פירוט מודולים

M1 · PROFILES — פרופילים ותחומי עניין

אחריותניהול פרופילי בני משפחה, תחומי עניין, מקורות, העדפות
קלטעדכוני משוב (M6), עריכה ידנית (M7/Cowork), social media enrichment
פלטfamily.json, scan_rules[], interest_vectors[]
אחסוןprofiles/ dir + member_preferences_log table
טכנולוגיהJSON config + SQLite log + optional vector embeddings
תחזוקת פרופילים: כל שינוי נשמר כ-snapshot ב-preferences_log עם סיבה (feedback/manual/auto). מאפשר rollback ומעקב אחרי אבולוציית העדפות.

M2 · SCANNER — סריקה ואיסוף

אחריותסריקת מקורות מידע, איסוף תכנים חדשים, deduplicate ראשוני
קלטscan_rules[] (מ-M1), sources.json (configurable)
פלטNCI[] — רשימת Normalized Content Items
סוגי מקורותRSS feeds, Web scraping (Playwright), YouTube API, WhatsApp incoming
עדכון מקורותPhase 1: עריכת sources.json דרך סשן Cowork + git push
Phase 2: דרך Dashboard UI (הוספה/הסרה/עריכה)
טכנולוגיהPython: feedparser, httpx, playwright, yt-dlp
גמישות מקורות: sources.json מגדיר כל מקור עם type, URL, schedule, trust_score, linked_members. שינוי ב-JSON = שינוי בסריקה. לא צריך לגעת בקוד.

M3 · NORMALIZER — עיבוד, ניקוד ואחסון

אחריותניקוד רלוונטיות, סינון, תמהיל, שילוב ידיעות משפחתיות, יצירת סיכומים, שמירה לארכיון
קלטNCI[] (מ-M2), submissions[] (מ-M6), profiles (מ-M1)
פלטNEO{} — Normalized Edition Object, מוכן לרינדור
תת-תהליכים 1. Dedup: hash + semantic similarity vs 30-day archive
2. Score: keyword match × priority × source trust × freshness × engagement history
3. Merge Submissions: שילוב ידיעות משפחתיות (חובת פרסום, עריכת AI)
4. Curate: top-N per member, topic balance, discovery selection
5. Generate: Claude API → headlines, summaries, trivia, greeting, edit submissions
6. Track Tokens: log all API calls with input/output tokens + cost
7. Archive: save all to SQLite + JSON daily snapshot
טכנולוגיהPython + Anthropic Claude API + SQLite + optional: sentence-transformers for embeddings
עריכת ידיעות משפחתיות: האיגנט פועל כעורך עיתון מקצועי — מסכם, מנסח כותרת, מארגן תמונות/מדיה, מוודא איכות. הידיעה תמיד מתפרסמת בגליון הקרוב.

M4 · RENDERER — רינדור HTML

אחריותהפיכת NEO לקובץ HTML מוכן, כולל CSS inline, OG tags, tracking
קלטNEO{}
פלטnewsletter-YYYY-MM-DD.html + og-image.png
תבניתJinja2 template (Style 4 - light clean, מאושר)
סקשן חדש"ידיעות מהמשפחה" — סקשן ייעודי לתכנים שנשלחו ע"י בני משפחה
טכנולוגיהJinja2 + CSS inliner + OG meta generation

M5 · DISTRIBUTOR — הפצה

אחריותהעלאת HTML לשרת nimrod.bio (FTP) + שליחת הודעות WhatsApp מותאמות
קלטnewsletter.html, member_phones[]
פלטpublic_url (nimrod.bio/newsletter/YYYY-MM-DD/), send_log[]
שרתFTP upload לתיקיית משנה ב-nimrod.bio (upress.co.il hosting)
WhatsAppTwilio WhatsApp Business API / Meta Cloud API (template messages)
שעת שליחה12:00 IST
טכנולוגיהPython: ftplib (FTP to upress), twilio
הודעת WhatsApp מותאמת: כל בן משפחה מקבל הודעה שונה עם הכותרות הרלוונטיות לו. שקד מקבל באנגלית. הלינק מוביל ל-nimrod.bio/newsletter/.

M6 · FEEDBACK + SUBMISSIONS — משוב וידיעות

אחריותקליטת משוב + קליטת ידיעות מבני המשפחה + עיבוד + למידה + עדכון פרופילים
ערוצי קלט (משוב) פסיבי: click tracking (UTM), read pixels, time-on-page
אקטיבי-קל: emoji rating בניוזלטר, WhatsApp survey (1-4)
אקטיבי-עמוק: סקר שבועי/חודשי
ערוצי קלט (ידיעות) WhatsApp: כל בן משפחה שולח הודעה לבוט
סוגי תוכן: טקסט, תמונה, וידאו, קישור, PDF
עיבוד: זיהוי סוג, download media, AI editing, confirmation reply
פלטProfile adjustments → M1, submission_items[] → M3, reports → M7
WebhookFastAPI endpoint (on home server) for incoming WhatsApp messages
טכנולוגיהPython FastAPI webhook + SQLite + Claude for text analysis + media storage
תהליך ידיעה משפחתית:
1. בן משפחה שולח הודעה לבוט WhatsApp (טקסט/תמונה/וידאו/PDF/קישור)
2. M6 מזהה סוג, מוריד מדיה, שומר ב-DB כ-FamilySubmission
3. שולח תשובה אוטומטית: "התקבל! יפורסם בגליון הקרוב 📰"
4. בזמן Build: M3 שולף submissions ממתינות, Claude עורך כעיתונאי מקצועי
5. הידיעה נכנסת לסקשן "ידיעות מהמשפחה" בגליון — חובת פרסום

M7 · DASHBOARD — בקרה ותובנות

אחריותכלי בקרה ומקום לקבל תובנות לשיפור ושדרוג המערכת. תמיכה בתהליך הפיתוח. משמש את המפתח (Claude) ואת המנהל (נימרוד) באותה מידה.
יכולות בקרה - צפייה בכל הניוזלטרים שנוצרו
- עריכת פרופילים והעדפות
- הוספה/הסרה/עריכה של מקורות (M2)
- שינוי תזמונים (Phase 2)
- Preview לפני שליחה
- Manual override: הוספת/הסרת תוכן מגליון
תובנות ודוחות - עלות טוקנים: יומית + חודשית (input tokens, output tokens, total cost USD)
- דשבורד: ratings, clicks, trends
- זיהוי אנומליות (מקור תקוע, ירידת engagement)
- המלצות לשיפור (AI-generated insights)
- סטטוס בריאות מערכת
טכנולוגיהWeb app: React/Svelte + FastAPI backend (or simple HTML dashboard)
מעקב עלויות: כל קריאת Claude API מתועדת ב-token_usage table עם: תאריך, מודול, model, input_tokens, output_tokens, cost_usd. Dashboard מציג סיכום יומי + חודשי + trend + התראה אם חורגים מתקציב.

7 הארכיון כנכס — מודל הנתונים

כל פריט שנאסף, כל ניוזלטר שנוצר, כל משוב, כל ידיעה משפחתית — נשמר לצמיתות. הארכיון הופך עם הזמן לזיכרון משפחתי.
┌─────────────────────────┐ │ content_archive │ ← כל פריט שנאסף אי פעם │ (5,400+ items/year) │ └────────────┬────────────┘ │ ┌─────────────────────────┼─────────────────────────┐ │ │ │ ┌────▼────────────┐ ┌────────▼────────┐ ┌────────────▼────┐ │ newsletters │ │ newsletter_items │ │ feedback │ │ (365/year) │ │ (item-member │ │ (ratings, │ │ │ │ mapping) │ │ clicks, text) │ └────────┬────────┘ └─────────────────┘ └────────┬────────┘ │ │ │ ┌─────────────────────┐ │ │ │ family_submissions │ ← ידיעות מבני המשפחה │ │ (media, text, PDF) │ (נשמרות לצמיתות) │ └─────────────────────┘ │ │ │ └──────────────────┬───────────────────────┘ │ ┌─────────▼──────────┐ │ preferences_log │ ← snapshots of evolving interests │ (trends over time) │ └────────────────────┘ │ ┌─────────▼──────────┐ │ token_usage │ ← עלויות יומיות וחודשיות │ (cost tracking) │ └────────────────────┘ QUERY EXAMPLES: ───────────────── "What interested Nimrod last winter?" → SELECT * FROM newsletter_items ni JOIN content_archive ca ON ni.content_id = ca.id WHERE ni.member_id = 'nimrod' AND ni.newsletter_date BETWEEN '2025-12-01' AND '2026-02-28' "Which sources get the most clicks?" → SELECT ca.source_name, COUNT(f.id) as clicks FROM feedback f JOIN content_archive ca ON f.value = ca.url WHERE f.type = 'click' GROUP BY ca.source_name ORDER BY clicks DESC "How have Tzlil's interests evolved?" → SELECT date, preferences_snapshot FROM member_preferences_log WHERE member_id = 'tzlil' ORDER BY date "How much did we spend on tokens this month?" → SELECT SUM(cost_usd) as total_cost, SUM(input_tokens) as total_input, SUM(output_tokens) as total_output FROM token_usage WHERE timestamp >= date('now', 'start of month') "What did the family share this year?" → SELECT fs.member_id, COUNT(*) as submissions FROM family_submissions fs WHERE fs.status = 'published' GROUP BY fs.member_id
