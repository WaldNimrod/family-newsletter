1 System Behavior Overview

תיאור התנהגות ברמת component — לא קוד, אלא מה המערכת עושה.

1.1 Daily Edition Lifecycle

כל יום המערכת מבצעת מחזור מלא: איסוף תוכן ממקורות חיצוניים, שילוב ידיעות שנשלחו ע"י בני המשפחה, ניקוד ותמהיל לפי פרופילים אישיים, יצירת סיכומים בעברית/אנגלית, רינדור ל-HTML, העלאה לשרת, והפצה ב-WhatsApp (או אימייל כ-fallback). בסוף היום נשלח סקר WhatsApp קצר עם שאלות מגוונות.

עיקרון מנחה — איכות תוכן ועיצוב קודם לכל: רוב המאמץ מושקע ב-M2 (מקורות איכותיים), M3 (סיכומים חכמים), ו-M4 (עיצוב מרשים). שאר המודולים פשוטים ככל האפשר.

1.2 Component-Level Behavior

Component מתי פועל התנהגות כשל
M1 Profiles Passive — data read on demand מספק פרופילים ו-scan_rules לכל מודול שמבקש. מתעדכן רק כשנכתב אליו (מ-M6, Cowork, Dashboard). אם family.json corrupt — abort pipeline, alert admin
M2 Scanner CRON 09:00 IST עובר על sources.json, מפעיל fetcher מתאים (RSS/Web/YouTube), מייצר NCI[] לכל מקור. דילוג על מקור שנכשל, המשך לבאים. מקור בודד נכשל: log + skip. כל המקורות נכשלו: abort + alert. Timeout per source: 30s.
M3 Normalizer Immediately after M2 מקבל NCI[] + pending submissions. מבצע: dedup → score → merge submissions → curate → generate (Claude API) → archive. מייצר NEO{}. Claude API fail: retry 3x with backoff. After 3: use title-only mode (no summaries). DB write fail: abort + alert.
M4 Renderer Immediately after M3 מקבל NEO{}, מפעיל Jinja2 template, מייצר HTML עם inline CSS. שומר locally. Template error: abort + alert. Output validation: HTML must be > 1KB.
M5 Distributor CRON 12:00 IST 1. FTP upload ל-nimrod.bio (חובה). 2. בדיקת URL live. 3. שליחת WhatsApp per member (או email כ-fallback). כל הודעה מותאמת אישית. FTP fail: retry 3x — חוסם, בלי FTP אין הפצה. WhatsApp fail: fallback to email. Email fail: log + alert.
M6 Feedback (Phase 1) 24/7 webhook + CRON 21:00 Webhook: קולט הודעות WhatsApp — מזהה submissions (ידיעות) vs. תשובות לסקר. 21:00: שולח סקר WhatsApp יומי עם שאלות מגוונות ורלוונטיות לתוכן היום. Phase 1 לא כולל: click tracking, pixels, UTM, analytics אוטומטי. Webhook down: WhatsApp will retry (up to 72h). Survey fail: skip, try tomorrow.
M7 Dashboard (Phase 2) On-demand (web UI) רץ על השרת הביתי או מחשב הפיתוח בלבד. לא על שרת nimrod.bio. מציג: newsletters, feedback, token costs, source health. מאפשר config changes. Read-only mode if DB locked. Config changes validated before save.

3.2 Family Submission State Machine

WhatsApp M6 saves M3 build picks up M3 publishes message in to DB and edits via Claude in edition │ │ │ │ ▼ ▼ ▼ ▼ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ INCOMING │──▶│ RECEIVED │──────────▶│ EDITING │──────────▶│PUBLISHED │ └──────────┘ └──────────┘ └──────────┘ └──────────┘ │ │ │ invalid sender │ Claude API fail ▼ ▼ ┌──────────┐ ┌──────────┐ │ REJECTED │ │EDIT_FAILED│───── retry next build ──▶ EDITING └──────────┘ └──────────┘ States stored in: family_submissions.status column Valid values: received | editing | published | rejected | edit_failed
Business Rule — Mandatory Publish: Every valid submission (status=received from a known member) MUST reach PUBLISHED state. If EDIT_FAILED, the system retries in the next daily build. A submission may never be silently dropped. If it fails 3 consecutive builds, alert admin for manual resolution.

4 Interface Contracts

כל ממשק בין components מוגדר: מי מייצר, מי צורך, מה עובר.
InterfaceProducerConsumerContractFormat
I-01: Scan Rules M1 Profiles M2 Scanner sources.json + family.json → list of sources with member linkage, keywords, priority, schedule JSON files on disk
I-02: Raw Items M2 Scanner M3 Normalizer NCI[] — array of Normalized Content Items. Each must have: id, url, title, source, published_at, language, content_hash. Empty array is valid (no new content). Python list of dataclass instances (in-process)
I-03: Pending Submissions M6 (DB) M3 Normalizer All rows from family_submissions where status IN ('received', 'edit_failed'). M3 reads directly from SQLite. SQLite query result
I-04: Edition Object M3 Normalizer M4 Renderer NEO{} — complete edition object. Must include: date, greeting, member_sections (with articles), family_content, discovery, trivia, metadata (including token usage). Python dataclass instance (in-process)
I-05: Newsletter HTML M4 Renderer M5 Distributor HTML file path on local disk. File must: exist, be > 1KB, contain valid HTML with <!DOCTYPE html>. Phase 2: include tracking pixel URL. File path string
I-06: FTP Upload M5 Distributor nimrod.bio server FTP PUT of HTML + media files to /newsletter/YYYY-MM-DD/. Server returns success/failure. FTP protocol
I-07: WhatsApp/Email Out M5 Distributor WhatsApp Business API / SMTP Primary: WhatsApp template message per member. Fallback: Email with same content (HTML email with link). If WhatsApp not configured yet → email only. See §4.2. HTTP POST to Twilio/Meta API — or — SMTP send
I-08: WhatsApp In WhatsApp Business API M6 Webhook Incoming messages (text, image, video, document, location). Webhook must respond 200 within 5s. See §4.3. HTTP POST to FastAPI endpoint
I-09: Profile Adjustments (Phase 2) M6 Feedback M1 Profiles Phase 2 only. JSON patch: {member_id, adjustments: [{topic, score_delta, reason}]}. Phase 1: profile changes are manual (Cowork session). Python dict (in-process) → JSON file write
I-10: Dashboard Read SQLite DB M7 Dashboard Read-only access to all tables. Dashboard never writes directly to operational tables — only to config files. SQLite queries via FastAPI backend
I-11: Claude API M3 Normalizer / M6 Feedback Anthropic API See §4.4 for all Claude API call patterns, models, token limits. HTTP POST to api.anthropic.com

4.1 NCI Full Schema (I-02)

NCI { id: string // SHA256(url)[:16] — deterministic, stable url: string // required, must be valid URL or "submission://{submission_id}" title: string // required, 1-500 chars source: { name: string // e.g. "Yachting World", "יויו (submission)" type: enum // "rss" | "web" | "youtube" | "api" | "family_submission" url: string // source home URL trust_score: float // 0.0–1.0, default 0.7 for external, 1.0 for family } published_at: ISO8601 // original publish date, or submission timestamp fetched_at: ISO8601 // when scanner fetched it language: enum // "he" | "en" — detected or from source config raw_text: string // summary/excerpt, 0-5000 chars tags: string[] // extracted or configured tags image_url: string? // nullable, must be valid URL if present media: { // nullable, for submissions with media type: enum // "image" | "video" | "pdf" local_path: string // path on home server after download original_url: string // WhatsApp media URL (temporary) mime_type: string size_bytes: int }? content_hash: string // SHA256(title + raw_text)[:16] — for dedup is_submission: boolean // true if from family member submitted_by: string? // member_id if is_submission=true }

Extract sender phone → match to family.json member_id - No match → ignore (log as unknown sender) 2. Check context: was a survey sent to this member in last 6 hours? - YES and message is short (< 50 chars or numeric) → route to SURVEY RESPONSE handler - NO or message is long / has media → route to SUBMISSION handler 3. Survey Response handler: - Save to feedback table (type="survey", value=message text) 4. Submission handler: - text only → save as text submission - media (image/video/PDF) → download media, save file, create submission - URL detected in text → fetch URL metadata, save as link submission

חודשי: ×30 = ~$2.50-3.50.

6 Business Rules

כל הכללים העסקיים שמנהלים את זרימות המערכת.

6.1 Content Scoring (M3)

Rule IDRuleFormula / Logic
BR-01 Relevance Score score = keyword_match × priority_weight × source_trust × freshness × engagement_bonus
BR-02 Priority Weights HIGH = 1.0, MEDIUM = 0.6, LOW = 0.3
BR-03 Freshness Decay Items > 48h old: score × 0.5. Items > 7d: score × 0.1. Items > 30d: excluded.
BR-04 Source Trust Initial: 0.7 (external), 1.0 (family). Weekly adjustment ±0.1 based on click rates.
BR-05 Engagement Bonus Articles from sources with > 50% click rate in last 14 days get +0.2 bonus.
BR-06 Deduplication content_hash match in last 30 days → exclude. Semantic similarity > 0.85 (if embeddings enabled) → exclude.
BR-07 Language Routing Shaked: EN only. All others: HE preferred, EN acceptable. Mixed articles: use raw language.

6.2 Curation Rules (M3)

Rule IDRuleDetails
BR-10 Max Items Per Member Configurable per member in family.json. Default: 3-5. Top-N by score after filtering.
BR-11 Total Edition Cap Max 15 unique articles per edition (across all members). Shared articles counted once.
BR-12 Discovery Ratio 20% of each member's items should be "discovery" — content from another member's domain, with bridge text.
BR-13 Topic Balance No more than 60% of a member's items from a single topic. Force diversity.
BR-14 Family Submission Override Submissions always included, regardless of score. They don't count toward the per-member cap. They appear in a dedicated "ידיעות מהמשפחה" section.
BR-15 Tzlil Puzzle Every edition MUST include a math puzzle for צליל. Difficulty: hard (competition level for age 13-14). Include yesterday's answer.

6.3 Submission Rules (M6 + M3)

Rule IDRuleDetails
BR-20 Mandatory Publish Every valid submission MUST be published in the next edition. No filtering, no scoring. Only AI editing.
BR-21 Known Sender Only Only messages from phone numbers in family.json are accepted. Unknown senders: ignored + logged.
BR-22 Media Size Limit Images: max 16MB. Video: max 64MB. PDF: max 32MB. Over limit: reject with friendly message.
BR-23 Editorial Processing Claude acts as professional newspaper editor: creates headline, writes summary, maintains original intent. Never changes meaning.
BR-24 Retry on Edit Failure If Claude API fails during editing: status → edit_failed, retry_count++. Retried next build. After 3 failures: alert admin, publish with raw text (no AI editing).
BR-25 Cutoff Time Submissions received after 08:30 IST go to next day's edition. Before 08:30 → today's edition.

6.4 Distribution Rules (M5)

Rule IDRuleDetails
BR-30 Send Time 12:00 IST. Configurable in settings.json.
BR-31 FTP Before WhatsApp Always upload HTML first, verify URL is live, THEN send WhatsApp messages. Never send link to non-existent page.
BR-32 Personalized Messages Each member gets their own WhatsApp message with headlines relevant to them. Shaked in English.
BR-33 Partial Send OK If WhatsApp fails for one member, continue sending to others. Log failure per member.
BR-34 Email Fallback If WhatsApp is not configured yet or fails for a member → send email with same personalized content + newsletter link. Email is always available as channel.
BR-35 FTP is Blocking FTP upload is mandatory. If FTP fails after 3 retries, edition status → SEND_FAILED, no messages sent (no broken links). Alert admin.

6.5 Feedback Rules (M6)

Rule IDRulePhaseDetails
BR-40 Daily WhatsApp Survey Phase 1 Daily at 21:00. Varied questions generated by Claude based on today's content. Not a fixed rating scale. Max 1 survey per day per member.
BR-41 Survey Response Storage Phase 1 All survey responses stored in feedback table as free text. Used for manual review in Phase 1, automated analysis in Phase 2.
BR-42 Click Attribution (UTM) Phase 2 UTM params in newsletter links: ?utm_source=famely&utm_member={id}&utm_article={content_id}. Requires tracking endpoint on home server.
BR-43 Automated Learning Loop Phase 2 Weekly: Claude analyzes survey responses + clicks. Adjusts source trust_scores (±0.1 max), suggests topic priority changes.
BR-44 Monthly Conversation Phase 1 Recommended (not blocking): conversation session with at least one family member. Manual process — no automation needed.
BR-45 Profile Changes (Phase 1) Phase 1 Profile changes are manual only — via Cowork session editing family.json. Every change saved as snapshot. Automated adjustments in Phase 2.

6.6 Token Cost Rules (M3 + M7)

Rule IDRuleDetails
BR-50 Log Every Call Every Claude API call → row in token_usage with exact input/output token counts and calculated USD cost.
BR-51 Daily Budget Alert If daily cost exceeds configurable threshold (default $0.50): log warning. Never block pipeline.
BR-52 Monthly Budget Alert If monthly cost exceeds configurable threshold (default $10): alert admin via WhatsApp.
BR-53 Dashboard Display Dashboard shows: today's cost, MTD cost, daily trend chart, cost per module breakdown.

8 Open Design Questions (Resolved)

שאלות עיצוב שנפתרו במהלך האפיון. חלק מהשאלות עדיין פתוחות — מסומנות בהתאם.
QuestionDecisionRationale
Where does server-side logic run? Home server only. nimrod.bio serves static files. Dashboard on home server or dev machine. upress.co.il hosting doesn't support server-side processes. Production hosting stays lean — no DB, no logic.
Is click tracking / analytics worth the complexity in Phase 1? No. Deferred to Phase 2. Static hosting on nimrod.bio can't run tracking endpoints. WhatsApp survey provides sufficient Phase 1 feedback. Simplicity > completeness. Focus Phase 1 on content quality and design.
What if WhatsApp API is complex or slow to set up? Email as fallback from day 1. WhatsApp can be added later. FTP is the only mandatory channel. Distribution must work. Email is universally available, zero approval process.
How are M2 sources updated? Phase 1: Cowork session edits sources.json + git push. Phase 2: Dashboard UI. Avoids building UI early. JSON config is simple and version-controlled.
How are submissions distinguished from feedback? Numeric reply 1-4 during active survey → feedback. All other messages → submission. Simple routing rule. No ambiguity.
What happens when Claude API is down? Retry 3x with exponential backoff. Then: title-only mode (no summaries). Newsletter must go out. Degraded quality is better than no newsletter.
Send time? 12:00 IST, configurable. User requirement. Build starts at 09:00 to allow 3h processing window.
Submission cutoff? 08:30 IST. After that → next day. Allows 30 min before build starts at 09:00.
Database engine? SQLite single file. No external DB server. Simple, portable, zero config. Sufficient for family-scale data.
How to handle source degradation? 3 failures → degraded (trust -0.2). 10 total → disabled. Manual reactivation. Auto-healing for transient issues. Manual intervention for dead sources.
